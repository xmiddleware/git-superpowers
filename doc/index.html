How Git Really Works

Think of Git like a photo album or game saves

üì∏ Snapshots, Not Diffs

When you commit, Git saves the ENTIRE state of your project‚Äîlike taking a complete photo, not just noting what changed.

üíæ Commits = Game Saves

Each commit is a complete snapshot. Like saving a game‚Äîyou get the full state, not just "what changed since last save."

In plain English: Every commit contains your entire project at that moment.

üìç Branches = Bookmarks

Branches are just pointers‚Äîlike bookmarks in a book. They don't copy anything, they just mark a spot.

In plain English: Creating a branch costs nothing. It's just a label.

üëÜ HEAD = Where You Are

HEAD is simply "where you are right now." Like your current location on a map‚Äîit can move freely.

In plain English: HEAD points to the commit you're currently looking at.

üìú Reflog = Your Safety Net

Reflog records every time HEAD moved. Like a security camera‚Äîit remembers everything, even if you "delete" it.

In plain English: Reflog is your backup. Nothing is ever truly lost.

SPEAKER NOTE: This is crucial. Many developers don't truly understand snapshots vs. diffs. Explain: "When you commit, Git saves the ENTIRE state of your project, not just changes." Use analogies: "Like saving a game. Each save is complete, not just what changed." "Branches are like bookmarks‚Äîthey don't copy pages, they just mark spots." Reality: Reset moves HEAD. The commits stay in reflog forever.

‚ùå Detached HEAD Is Unsafe

Reality: Detached HEAD is a playground. Create a branch to keep work.

Common Confusions‚ÄîClarified

Simple answers to questions students often ask

‚ùì "Branches copy files"

No! Branches are just pointers‚Äîlike bookmarks. They don't copy anything. Creating a branch is instant and free.

‚ùì "Deleted commits are gone forever"

No! Reflog keeps them for 90 days. Even if you delete a branch, commits persist in reflog. Nothing is truly lost.

‚ùì "Detached HEAD is dangerous"

No! It's a safe playground. Explore freely. If you want to keep work, create a branch. If not, just checkout main.

‚ùì "Reset and revert do the same thing"

No! Reset rewrites history (local only). Revert creates a new commit that undoes changes (safe for shared branches).

‚ùì "I need to understand everything before using Git"

No! Experiment freely. Git protects you. Reflog is your safety net. Branches isolate experiments. Nothing is permanent until you push.

‚ùì "Git is complicated and scary"

No! Git rewards experimentation. It punishes nothing. Once you understand reflog and branches, fear disappears.

The Pattern: Most Git fears come from misunderstanding. Once you know the truth, Git becomes your friend.

SPEAKER NOTE: Address these directly. Many students hold these misconceptions. Clear them up now before they cause anxiety. Say: "Raise your hand if you thought branches copy files." Normalize the confusion. Then explain clearly.

Superpower #3a: Reset

Local History Rewriting

Use when: You want to undo commits on a LOCAL branch (not shared with teammates)

git reset --soft HEAD~1

Moves HEAD back, keeps changes staged

git reset --mixed HEAD~1

Moves HEAD back, keeps changes unstaged (default)

git reset --hard HEAD~1

Moves HEAD back, discards all changes

Reset Flow
main ‚Üí commit3 ‚Üí commit2 ‚Üí commit1
‚Üì git reset --hard HEAD~1
main ‚Üí commit2 ‚Üí commit1
(commit3 still in reflog!)

‚ö†Ô∏è Remember: Only use reset on LOCAL branches. Never on shared branches (main, develop).

SPEAKER NOTE: "Reset rewrites history. That's fine on YOUR branch. But if teammates have that commit, reset breaks their history. That's why we have revert for shared branches." Common Questions: "What's the difference between --soft, --mixed, --hard?" ‚Üí "Soft keeps staged, mixed keeps unstaged, hard deletes everything."
Agile Actors

Superpower #3b: Revert

Safe Undo for Shared Branches

Use when: You need to undo a commit that's already on a SHARED branch (main, develop)

git revert [commit]

Creates a NEW commit that undoes the changes

History stays intact. Safe for teammates.

Revert Flow
main ‚Üí commit3 ‚Üí commit2 ‚Üí commit1
‚Üì git revert commit3
main ‚Üí commit4 (undoes commit3) ‚Üí commit3 ‚Üí commit2 ‚Üí commit1
(History preserved! Safe for shared branches)

‚úÖ Perfect for: Undoing commits on main, develop, or any branch teammates use.

SPEAKER NOTE: "Revert doesn't delete history. It adds a new commit that undoes changes. This way, teammates' history stays consistent. They can pull safely." Common Questions: "What if I revert a revert?" ‚Üí "You undo the undo. It works fine."
Agile Actors

Superpower #3c: Checkout

Discard Uncommitted Changes

Use when: You want to discard changes in UNCOMMITTED files (not yet committed)

git checkout [file]

Restores file to the state of the last commit

Only affects unstaged changes. Safe anytime.

BEFORE: Modified File

$ git status
modified: src/Main.java
$ cat src/Main.java
// Modified content
// That you want to discard

AFTER: Restored File

$ git checkout src/Main.java
$ git status

Key Point: Checkout only affects files. It doesn't change commit history.

SPEAKER NOTE: "Checkout is for uncommitted changes. If you haven't committed yet, checkout discards your modifications. Simple and safe." Common Questions: "What if I want to discard all files?" ‚Üí "git checkout . (but be careful!)"

Three Ways to Undo

Quick Quiz

1. When should you use reset vs revert vs checkout?
‚Üí Answer: Reset for LOCAL branches (rewrites history). Revert for SHARED branches (preserves history). Checkout for UNCOMMITTED file changes (no history impact).
2. What's the key difference between reset and revert?
‚Üí Answer: Reset moves HEAD and rewrites history (safe on local only). Revert creates a new commit that undoes changes (safe for shared branches).
3. Can you reset on a shared branch like main?
‚Üí Answer: Technically yes, but DON'T. Reset rewrites history, which breaks teammates' local branches. Use revert on shared branches instead.
4. What happens if you checkout a file that has uncommitted changes?
‚Üí Answer: Checkout discards the uncommitted changes and restores the file to the state of the last commit. The changes are lost unless you stash or commit them first.
5. If you reset a commit, is it truly deleted?
‚Üí Answer: No! The commit is removed from the branch history, but it remains in reflog. You can recover it using `git reflog` and `git reset --hard [hash]`.

Local branches: reset OK. Shared branches: use revert. Uncommitted changes: use checkout. Choose the right tool for the context.

Common Mistakes & Recoveries

What NOT to do, and how to fix it

‚ùå Mistake #1: Force Push to Shared Branch

Problem: You force pushed to main/develop. Teammates' local branches are now inconsistent.
Recovery: Teammates can recover with their reflog. Coordinate carefully. Consider reverting instead.
Prevention: Never force push to shared branches. Use revert for shared branches.

‚ùå Mistake #2: Reset on Shared Branch

Problem: You reset main locally. When you push, teammates' history breaks.
Recovery: Use reflog to restore. Then use revert instead. Coordinate with team.
Prevention: Only reset on local branches. Use revert for shared branches.

‚ùå Mistake #3: Committing on Detached HEAD Without Branching

Problem: You checked out an old commit, made changes, committed. Then checked out main. Work is "lost".
Recovery: Use reflog to find the commit. Create a branch from it: git checkout -b recovery-branch [hash]
Prevention: Always create a branch when committing in detached HEAD: git checkout -b new-branch

‚ùå Mistake #4: Deleting Branch Before Merging

Problem: You deleted a branch thinking the work was lost.
Recovery: Commits are in reflog! Find the commit hash and recreate: git checkout -b recovered [hash]
Prevention: Remember: deleting a branch only deletes the pointer. Commits persist in reflog.

Remember: Most mistakes are recoverable with reflog. Don't panic. Git has your back.

SPEAKER NOTE: "These are real mistakes developers make. The good news: Git can recover from almost all of them. The key is knowing HOW." Common Questions: "What if I can't find it in reflog?" ‚Üí "Check if it's been more than 90 days. Otherwise, it should be there."

Professional Safety Rules

When rewriting history is safe. When it's not.

‚úÖ SAFE: Rewrite on LOCAL/PERSONAL Branches

  • Reset, rebase, force push‚Äîall OK
  • No teammates are pulling from it
  • Only you see the history

‚ö†Ô∏è CAREFUL: NEVER Rewrite on SHARED Branches

  • Reset, rebase, force push = teammates' pain
  • Their local branches become inconsistent
  • Use revert instead (creates new commit, history intact)
SPEAKER NOTE: "The golden rule: You can break your own house. You cannot break your teammate's house. Local branches are yours. Shared branches (main, develop) are everyone's. Revert there, not reset." Common Questions: "How do I know if a branch is shared?" ‚Üí "If others might pull from it, it's shared. When in doubt, assume it's shared."
