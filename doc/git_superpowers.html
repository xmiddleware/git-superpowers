How Git Really Works

Snapshots, Not Diffs

  • Commits are immutable snapshots of your entire project at a point in time
  • Branches are just pointers to commits‚Äîthey're cheap and safe to create
  • HEAD is simply "where you are right now"‚Äîit can move freely
  • The reflog records every movement of HEAD‚Äîyour safety net
SPEAKER NOTE: This is crucial. Many developers don't truly understand snapshots vs. diffs. Explain: "When you commit, Git saves the ENTIRE state of your project, not just changes." Then: "Live demo time. Watch as I create the project from scratch."

Superpower #1: Git Reflog

Absolute Recovery Tool

What it is: A log of every time HEAD moved. Your commit history backup.

  • Even if you "lose" a commit with reset, it's in reflog
  • Even if you delete a branch, commits are in reflog
  • Reflog is LOCAL‚Äîonly your computer tracks it
  • NOTHING is ever truly deleted once committed
SPEAKER NOTE: "This is the most powerful tool. With reflog, you literally cannot lose work. You can intentionally break things because recovery is guaranteed. Now we demo this."

Superpower #2: Detached HEAD

Safe Exploration Playground

What it is: Checking out an old commit without being on a branch. Pure experimentation space.

  • Explore old code without fear
  • Test if a bug existed at commit X
  • If you make changes: create a branch to save them
  • If you discard changes: just checkout main‚Äîno damage
SPEAKER NOTE: "Detached HEAD sounds scary. It's actually liberating. You're not on a branch‚Äîyou're exploring a point in time. Want to see if a bug existed 5 commits ago? Detached HEAD. Want to understand old code? Detached HEAD. And it's 100% safe."

Superpower #3: Three Ways to Undo

Same mistake. Three different recovery tools.

git reset [commit]

Use when: You want to move HEAD and undo commits (locally only)

Modes: --soft (keep changes), --mixed (staged changes), --hard (delete changes)

git revert [commit]

Use when: You want to UNDO a commit's changes (safe for shared branches)

Creates NEW commit that undoes the changes‚Äîhistory stays intact

git checkout [file]

Use when: You want to discard changes in UNCOMMITTED files

Restores file to last commit‚Äîonly affects unstaged changes

SPEAKER NOTE: "These three are confused constantly. Reset rewrites history (local). Revert adds a new commit (safe for shared code). Checkout discards unstaged changes (file level). I'll demo all three fixing the SAME mistake‚Äîthen you'll know when to use each."

Demo: Three Undo Strategies

Fixing the same mistake three different ways

Strategy 1: Reset (Rewrite History Locally)

Move HEAD Back

Superpower #4: Branches Are Cheap

Experiment Without Risk

Key Truth: A branch is just a pointer to a commit. Create as many as you want. Delete without guilt.

  • Create experimental branches for any idea
  • Break code intentionally‚Äîit's isolated
  • Delete the branch if the idea fails
  • The commits stay in reflog‚Äînothing is lost
SPEAKER NOTE: "Branches cost nothing. They're pointers. Want to test an idea? Branch. Break code? No problem. Discard? Delete. The commits are still in reflog. This is liberation."

Superpower #5: Stash

The Panic Button

What it is: Temporarily save uncommitted work without committing. Perfect for context switching.

  • Emergency hits: stash your work, switch branches, fix emergency
  • Work is saved‚Äînothing is lost
  • Come back later and restore with git stash pop
  • Safe, temporary, pressure-free
SPEAKER NOTE: "Stash is psychological relief. You're halfway through code. Emergency happens. Instead of panicking, stash. It's safe. You come back. It's still there. No pressure."
Agile Actors

Superpower #6: Cherry-pick

Surgical Commit Recovery

What it is: Pick ONE specific commit from anywhere in history and apply it to your current branch.

  • You made a good commit on the wrong branch
  • You want one commit from experimental branch
  • Cherry-pick grabs it and applies it here
  • Clean. Surgical. Intentional.
SPEAKER NOTE: "Cherry-pick is for selective, intentional recovery. You don't want the whole branch‚Äîjust that ONE commit. It's precise. Powerful. Professional."

Professional Safety Rules

When rewriting history is safe. When it's not.

‚úÖ SAFE: Rewrite on LOCAL/PERSONAL Branches

  • Reset, rebase, force push‚Äîall OK
  • No teammates are pulling from it
  • Only you see the history

‚ö†Ô∏è CAREFUL: NEVER Rewrite on SHARED Branches

  • Reset, rebase, force push = teammates' pain
  • Their local branches become inconsistent
  • Use revert instead (creates new commit, history intact)
SPEAKER NOTE: "The golden rule: You can break your own house. You cannot break your teammate's house. Local branches are yours. Shared branches (main, develop) are everyone's. Revert there, not reset."
